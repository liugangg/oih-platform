#### 4.2 QLoRA 配置
```python
# 4-bit 量化加载
BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",          # NormalFloat4
    bnb_4bit_compute_dtype=torch.bfloat16,  # 计算精度
    bnb_4bit_use_double_quant=True,      # 二次量化，节省 ~0.4GB
)

# LoRA adapter
LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,                                # LoRA rank
    lora_alpha=32,                       # scaling factor (alpha/r = 2)
    lora_dropout=0.05,
    target_modules=[                     # 覆盖全部 attention + MLP
        "q_proj", "k_proj", "v_proj", "o_proj",
        "gate_proj", "up_proj", "down_proj",
    ],
    bias="none",
)
```

**可训练参数**: 64,225,280 / 14,832,532,480 = **0.433%**

#### 4.3 训练参数
```python
SFTConfig(
    num_train_epochs=5,
    per_device_train_batch_size=2,
    gradient_accumulation_steps=4,       # effective batch = 8
    learning_rate=2e-4,
    lr_scheduler_type="cosine",
    warmup_ratio=0.1,
    weight_decay=0.01,
    max_grad_norm=0.3,
    bf16=True,
    max_length=2048,
    optim="paged_adamw_8bit",            # 8-bit AdamW 省内存
    gradient_checkpointing=True,
    save_strategy="epoch",
    eval_strategy="epoch",
    load_best_model_at_end=True,         # 自动选最佳 checkpoint
    metric_for_best_model="eval_loss",
    attn_implementation="sdpa",          # PyTorch 内置 Scaled Dot Product Attention
)
```

**训练步数**: 90 samples / (batch 2 × accum 4) = 12 steps/epoch × 5 epochs = **60 steps**

### 8. 部署方案

#### 方案 A: 替换当前 vLLM（推荐）
```bash
# 1. 停止当前 vLLM
kill $(pgrep -f "vllm.*Qwen3-14B-AWQ")

# 2. 启动微调版
NO_PROXY='*' no_proxy='*' nohup /data/oih/miniconda/bin/python -m vllm.entrypoints.openai.api_server \
    --model /data/oih/models/Qwen3-14B-OIH-AWQ \
    --served-model-name Qwen3-14B \
    --port 8002 --host 0.0.0.0 \
    --quantization awq --dtype auto \
    --max-model-len 32768 \
    --gpu-memory-utilization 0.90 \
    --enable-prefix-caching \
    --reasoning-parser deepseek_r1 \
    --enable-auto-tool-choice --tool-call-parser hermes \
    --trust-remote-code > /tmp/vllm.log 2>&1 &
```

#### 10.6 Agent 代码修复（5 个问题，逐步排查）

**问题 1: `message.get("content", "")` 返回 None**
- 根因: vLLM 返回 `content: null`（key 存在但值为 None），`.get("content", "")` 返回 None 而不是 ""
- 修复: `raw = (message.get("content") or "") or (message.get("reasoning") or "") or ""`

**问题 2: Tool calls 被 reasoning parser 截获**
- vLLM 返回 `tool_calls: []`，实际 tool call 在 `reasoning` 字段中以 `<tool_call>` 标签形式存在
- 根因: `--reasoning-parser deepseek_r1` 和 `--tool-call-parser hermes` 冲突，reasoning parser 先截获输出
- 修复: 从 `reasoning` 字段解析 `<tool_call>` 标签
```python
tc_matches = re.findall(r"<tool_call>\s*(.*?)\s*</tool_call>", reasoning_text, re.DOTALL)
```
- 同时修改终止条件: `if not message.get("tool_calls") and finish_reason == "stop"`（防止已解析的 tool calls 被跳过）

**问题 3: 简单问候触发工具调用死循环**
- v1 微调模型对 "你好" 反复调用 `get_system_status`（8 轮死循环）
- 根因: 训练数据全是 tool-calling 案例 + 系统提示包含完整工具列表
- 代码修复: `skip_skills=True` 时不传 `tools`、`tool_choice`、`extra_body`
- 数据修复: v2 新增 85 条非工具调用样本

**问题 4: `enable_thinking: true` + `thinking_budget=0` 导致空回复**
- 传 `extra_body: {enable_thinking: true, thinking_budget: 0}` 时 content=None, reasoning="\n\n"（全空）
- 修复: `skip_skills=True` 时完全不传 `extra_body`，让 vLLM 用默认行为

### 14. 任务取消修复

**问题**: 前端点取消后任务状态变为 cancelled，但容器内进程继续运行占用 GPU。

**根因**: `cancel_task()` 只修改数据库状态，不终止 `docker exec` 启动的容器内进程。

**修复**: `core/task_manager.py`

1. **`cancel_task()` 增加容器进程清理**:
```python
async def cancel_task(self, task_id: str) -> bool:
    ...
    task.status = TaskStatus.CANCELLED
    # Kill container processes if task was running
    if was_running and tool:
        await self._cleanup_container_after_task(tool)
    return True
```

2. **`_run()` 增加取消状态检查**:
```python
result = await coro_fn(task)
if task.status == TaskStatus.CANCELLED:
    logger.info("Task cancelled during execution, discarding result")
else:
    task.result = result
    task.status = TaskStatus.COMPLETED
```

`alphafold3, rfdiffusion, proteinmpnn, bindcraft, gromacs, diffdock, gnina, discotope3, igfold, esm`

每个工具映射到容器名 + 进程模式，通过 `docker exec kill -9` 终止。
